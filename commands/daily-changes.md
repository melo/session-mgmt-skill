Generate daily changelog documents from git history, with multi-day catch-up and a draft system for the current day.

Works in any git repository. Project-specific category mappings can be defined in `.claude/changelog-categories.yml`.

## Arguments

`$ARGUMENTS` — one of:
- **Empty** (catch-up mode): generate all missing days from the last finalized document through today
- **`YYYY-MM-DD`**: generate for a single date
- **`--from YYYY-MM-DD --to YYYY-MM-DD`**: generate for an explicit date range

## Steps

### 1. Run the preprocessing script

```bash
REPO_ROOT=$(git rev-parse --show-toplevel)
python3 ~/.claude/skills/session-mgmt-skill/scripts/collect_daily_changes.py $ARGUMENTS --repo-root "$REPO_ROOT"
```

This script handles ALL mechanical work:
- Collects git commits (numstat, stat, full messages)
- Reads session state.json files and computes three time metrics (daily span, active time, total work)
- Categorizes commits using `.claude/changelog-categories.yml` and heuristics
- Generates session timeline SVGs
- Formats sessions and commits tables
- Produces one `<yyyymmdd>-data.md` file per date in `docs/changes/<yyyy>/`

The script prints a JSON summary to stdout listing all generated data files.

### 2. Process each data file

For each data file generated by the script:

1. **Read the data file.** It has two parts separated by `<!-- REFERENCE DATA -->`:
   - **Part A (skeleton):** Near-complete markdown document with `<!-- PLACEHOLDER -->` markers where category summaries go.
   - **Part B (reference appendix):** Commit messages and diffstats grouped by category, for writing the summaries.

2. **For each `<!-- PLACEHOLDER -->` section,** write a narrative summary:
   - Read the `<!-- CATEGORY: ... -->` header to know the section topic.
   - Look at the corresponding reference data in Part B for that category's commits.
   - Write a `## Section Name` heading and a brief (2-4 sentence) description of what changed and why.
   - Use this style (from existing documents):
     ```markdown
     ## APIs — Trusted Caller Registration

     ### Trusted caller register endpoint with legacy user linking

     Extended the `/register` endpoint to support trusted-caller scenarios...
     ```
   - For categories with multiple sub-groups, use `### Sub-heading` per distinct feature.
   - Each section ends with `---`.

3. **Remove the reference appendix** (everything from `<!-- REFERENCE DATA -->` onwards).

4. **Determine the filename:**
   - Read the first `<!-- CATEGORY: ... -->` heading — the category with the most commits becomes the primary section.
   - Filename: `<yyyymmdd>-changes-<primary-section>.md` (or `-DRAFT.md` if the document header says "DRAFT").
   - `<primary-section>` is kebab-case, lowercase (e.g., "APIs" → "apis", "Backoffice" → "backoffice").

5. **Write the final document** to `docs/changes/<yyyy>/<filename>`.

6. **Delete the data file** (`<yyyymmdd>-data.md`).

### 3. Final summary

Print a summary of all documents generated:

| Date | File | Status |
|------|------|--------|
| 2026-02-20 | `2026/20260220-changes-apis.md` | Generated |
| 2026-02-21 | `2026/20260221-changes-apis-DRAFT.md` | Draft |
